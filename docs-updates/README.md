# docs-updates/

Proposed and applied documentation updates generated by `/sync-docs`.

## Files in this directory

| File | Description |
|---|---|
| `proposed-YYYY-MM-DD-[slug].md` | Proposed edits for review, one file per URL processed |
| `.applied.json` | Tracking log of all URLs processed and their status |

## Workflow

```
/sync-docs [URL]          →  creates proposed-YYYY-MM-DD-[slug].md (status: pending)
                              updates .applied.json
(you review the file)
/apply-docs [file]        →  applies edits to real docs
                              updates .applied.json (status: applied)
```

## Proposed-edits file format

Each file starts with a YAML front matter block, followed by `<!-- EDIT:N -->` blocks:

```
---
source_url: https://...
fetched: YYYY-MM-DD
status: pending | applied
edits_count: N
---

<!-- EDIT:1 -->
TARGET: path/to/file.md
AFTER_HEADING: ## Section Name
ACTION: insert_section | update_section | prepend_warning
REASON: why this doc is affected
---
[content to insert]
<!-- /EDIT:1 -->
```

## .applied.json format

```json
[
  {
    "url": "https://...",
    "fetched": "YYYY-MM-DD",
    "proposed_file": "docs-updates/proposed-2026-05-05-wordpress-5-8.md",
    "edits_count": 4,
    "edits_applied": 3,
    "edits_skipped": 1,
    "status": "pending | applied | skipped",
    "applied_at": "YYYY-MM-DD | null"
  }
]
```

## Safety guarantees

- `/apply-docs` never deletes existing content — inserts and prepends only
- Re-running `/sync-docs` on the same URL is blocked unless you pass `--force`
- Re-running `/apply-docs` on an already-applied file is a no-op
- Python `.py` files are flagged but never edited — all Python changes are manual
